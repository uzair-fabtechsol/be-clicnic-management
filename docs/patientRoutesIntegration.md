# Patient API — Frontend Integration Guide

Base path: `/api/v1/patients`

## Auth

All routes require the user to be signed in via **httpOnly cookie** `accessToken`. Send requests with `credentials: "include"` (fetch) / `withCredentials: true` (axios). Each route requires a permission on the `patients` module (`view` / `create` / `edit` / `delete`).

## Common response shape

```json
{
  "status": "success" | "fail" | "error",
  "message": "human readable message",
  "data": { ... } | null
}
```

## Common error responses

| Status | When |
|---|---|
| 400 | Zod validation failed (body/query) or invalid `:id` ObjectId |
| 401 | Not signed in / expired token |
| 403 | Missing required permission |
| 404 | Patient not found |
| 409 | CNIC already registered |

## 2. Create patient

`POST /api/v1/patients`

**Permission required:** `patients:create`

`mrNumber` and `registrationDate` are generated by the server — do not send them. MR numbers follow the sequential pattern `MR-YYMM####` (e.g. `MR-26070001`, `MR-26070002`, ...).

### Body

| Field | Type | Required | Notes |
|---|---|---|---|
| `name` | string | yes | trimmed, min 1 char |
| `guardianName` | string | yes | trimmed, min 1 char |
| `gender` | `"male" \| "female" \| "other"` | yes | |
| `age` | number | yes | min 1, max 150 |
| `ageUnit` | `"years" \| "months" \| "days"` | yes | |
| `mobileNumber` | string | no | Pakistani format only — see below |
| `cnic` | string | no | format `XXXXX-XXXXXXX-X` |
| `address` | string | no | trimmed, min 1 char |

**`mobileNumber` accepted format:** `03XX-XXXXXXX` only (e.g. `0331-9193087`) — `+92` / `0092` prefixes are no longer accepted. Send an empty string or omit the field entirely if the patient has no mobile number — both are treated as "not provided."

### Example request body

```json
{
  "name": "Ali Khan",
  "guardianName": "Ahmed Khan",
  "gender": "male",
  "age": 30,
  "ageUnit": "years",
  "mobileNumber": "0331-9193087",
  "cnic": "35202-1234567-1",
  "address": "Lahore, Pakistan"
}
```

### Example response — 201

```json
{
  "status": "success",
  "message": "Patient created successfully",
  "data": {
    "patient": {
      "_id": "64f1c2b3a1b2c3d4e5f6a7aa",
      "mrNumber": "MR-26070001",
      "name": "Ali Khan",
      "guardianName": "Ahmed Khan",
      "gender": "male",
      "age": 30,
      "ageUnit": "years",
      "mobileNumber": "0331-9193087",
      "cnic": "35202-1234567-1",
      "address": "Lahore, Pakistan",
      "registrationDate": "2026-07-16T10:00:00.000Z",
      "lastVisit": null,
      "createdAt": "2026-07-16T10:00:00.000Z",
      "updatedAt": "2026-07-16T10:00:00.000Z"
    }
  }
}
```

## 3. Bulk create patients

`POST /api/v1/patients/bulk`

**Permission required:** `patients:create`

Use this for importing existing/historical patient records. Send an **array** (1 to 100 items) of patient objects — same fields as [Create patient](#2-create-patient), plus a required `registrationDate` per item (this lets you backdate imported records instead of using "now"). MR numbers are still generated server-side, sequenced per year-month based on each item's own `registrationDate`.

This endpoint does **not** fail the whole batch if one item is invalid at the DB level (e.g. duplicate CNIC) — it processes every item and reports success/failure per index, so a partial success is possible. Malformed request bodies (wrong types, missing required fields) still fail the whole request with a 400, since that's caught by Zod before any item is processed.

### Body

Array of:

| Field | Type | Required | Notes |
|---|---|---|---|
| `name` | string | yes | trimmed, min 1 char |
| `guardianName` | string | yes | trimmed, min 1 char |
| `gender` | `"male" \| "female" \| "other"` | yes | |
| `age` | number | yes | min 1, max 150 |
| `ageUnit` | `"years" \| "months" \| "days"` | yes | |
| `registrationDate` | date string | yes | coerced via `z.coerce.date()` — ISO string recommended |
| `mobileNumber` | string | no | same Pakistani format rules as create |
| `cnic` | string | no | format `XXXXX-XXXXXXX-X` |
| `address` | string | no | trimmed, min 1 char |

### Example request body

```json
[
  {
    "name": "Ali Khan",
    "guardianName": "Ahmed Khan",
    "gender": "male",
    "age": 30,
    "ageUnit": "years",
    "registrationDate": "2025-03-10T09:00:00.000Z",
    "mobileNumber": "0331-9193087",
    "cnic": "35202-1234567-1",
    "address": "Lahore, Pakistan"
  },
  {
    "name": "Sara Ahmed",
    "guardianName": "Bilal Ahmed",
    "gender": "female",
    "age": 5,
    "ageUnit": "months",
    "registrationDate": "2025-03-11T09:00:00.000Z"
  }
]
```

### Example response — 201

```json
{
  "status": "success",
  "message": "Patients processed successfully",
  "data": {
    "results": [
      {
        "index": 0,
        "success": true,
        "patient": {
          "_id": "64f1c2b3a1b2c3d4e5f6a7aa",
          "mrNumber": "MR-25030001",
          "name": "Ali Khan",
          "...": "rest of patient fields"
        }
      },
      {
        "index": 1,
        "success": false,
        "error": "E11000 duplicate key error ... cnic ..."
      }
    ]
  }
}
```

Check `results[i].success` for each item — `true` entries have a `patient` object, `false` entries have an `error` message you can surface next to the corresponding row in your import UI.

## 5. Update patient

`PATCH /api/v1/patients/:id`

**Permission required:** `patients:edit`

### Body (all fields optional — send only what changes)

| Field | Type | Notes |
|---|---|---|
| `name` | string | trimmed, min 1 char |
| `guardianName` | string | trimmed, min 1 char |
| `gender` | `"male" \| "female" \| "other"` | |
| `age` | number (int) | min 1, max 150 |
| `ageUnit` | `"years" \| "months" \| "days"` | |
| `mobileNumber` | string | Pakistani format only — see [Create patient](#2-create-patient) |
| `cnic` | string | format `XXXXX-XXXXXXX-X` |
| `address` | string | trimmed, min 1 char |
| `lastVisit` | date string | normally set automatically when an OPD slip is created — only send this manually if you have a specific reason to override it |

### Example response — 200

```json
{
  "status": "success",
  "message": "Patient updated successfully",
  "data": {
    "patient": { "...": "updated patient object" }
  }
}
```

## 6. Delete patient

`DELETE /api/v1/patients/:id`

**Permission required:** `patients:delete`

No body. Hard-deletes the record.

### Example response — 200

```json
{
  "status": "success",
  "message": "Patient deleted successfully",
  "data": null
}
```

### 404

```json
{ "status": "fail", "message": "Patient not found", "data": null }
```
